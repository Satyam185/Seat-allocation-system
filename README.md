# Seat Allocation & Project Mapping System

MVP full-stack app for managing seat allocation and project mapping 
for ~5,000 employees. Built as a Next.js monorepo (frontend + API 
routes combined) instead of a separate FastAPI backend — see 
`DECISIONS.md` for why.

## Tech Stack
- **Framework:** Next.js (App Router) + TypeScript
- **Styling:** Tailwind CSS + shadcn/ui (Nova preset — Lucide icons + Geist font)
- **Database:** PostgreSQL via Supabase
- **ORM:** Prisma
- **Validation:** Zod
- **AI:** Groq / Gemini (NLQ search feature)
- **Deployment:** Vercel

## Setup

1. `npm install`
2. Copy `.env.example` → `.env` and fill in:
   - `DATABASE_URL` / `DIRECT_URL` — from your Supabase project (Settings → Database)
   - `GROQ_API_KEY` or `GEMINI_API_KEY` — pick one
3. `npx prisma generate`
4. `npx prisma migrate dev --name init`
5. `npx prisma db seed` — populates 5,000 employees, 500 projects, 6,000 seats
6. `npm run dev` → http://localhost:3000

## Project Structure & Folder Purpose

### `prisma/`
- `schema.prisma` — data model (Employee, Project, Seat, AllocationHistory + enums)
- `seed.ts` — faker-based seed script, deterministic (`faker.seed(42)`)
- `migrations/` — auto-generated by `prisma migrate dev`, do not edit manually

### `lib/`
- `db.ts` — Prisma client singleton. Always import `prisma` from here, never instantiate `new PrismaClient()` elsewhere (causes connection exhaustion in dev).
- `types.ts` — shared TypeScript types/enums used across frontend and backend (currently minimal — role type lives in `role-provider.tsx`, import from there instead of duplicating)
- `validations/` — Zod schemas, one file per entity (`employee.schema.ts`, `seat.schema.ts`, `project.schema.ts`, `ai.schema.ts`). These are the single source of truth for what a valid request payload looks like — reuse them in both API routes (server-side) and forms (client-side), don't write separate validation logic.
- `services/` — **business logic layer, currently empty stubs.** This is where real rules belong: seat assign/release/conflict handling, new-joiner allocation logic, project mapping rules, utilization/dashboard aggregation queries, and the AI NLQ → filter → query pipeline. API routes should stay thin (parse request → call service → return response); do not let business logic live inside `route.ts` files. Some routes currently have logic written inline as a placeholder — move that into the matching service file as you build out real rules.
  - `employee.service.ts`
  - `seat.service.ts`
  - `project.service.ts`
  - `analytics.service.ts`
  - `ai.service.ts`

### `app/api/`
REST endpoints. Each resource folder follows the same pattern:
- `route.ts` — GET (list, paginated) and POST (create)
- `[id]/route.ts` — GET one / PATCH / DELETE (not yet built for all resources — extend as needed)

Resources: `employees/`, `seats/`, `projects/`, `analytics/`, `ai/`, `seed-check/` (dev-only sanity check endpoint).

### `components/`
- `ui/` — shadcn/ui primitives (`button`, `table`, `select`, `card` installed; add more via `npx shadcn@latest add <component>` as needed — **use shadcn components for all UI elements, don't hand-roll basic inputs/buttons/tables**)
- `role-provider.tsx` / `role-switcher.tsx` — fake RBAC system (no real auth). Wrap conditional UI in `const { role } = useRole()` checks to show/hide actions per role (ADMIN, HR, EMPLOYEE, PROJECT_LEAD).
- `search-filters.tsx`, `seat-table.tsx`, `utilization-chart.tsx`, `new-joiner-form.tsx` — currently empty stubs, main UI build-out work

### Root files
- `DECISIONS.md` — documented assumptions and trade-offs (no real auth, 1:1 employee-seat, monorepo choice). Read this before assuming a requirement was missed — it may be an intentional scope decision.
- `AI_PROMPTS.md` — log of AI prompts used during development. **Keep updating this as you work** — paste real prompts and real outputs, not summaries, for every meaningful AI-assisted step (schema tweaks, service logic, AI NLQ prompt engineering, debugging).
- `.env.example` — required environment variables, copy to `.env` and fill in real values (never commit `.env`)

## Rules for extending this codebase
- Every empty stub file listed above exists on purpose and is expected to be filled — do not leave any of them blank in the final submission.
- Business logic goes in `lib/services/`, never directly in `app/api/*/route.ts`.
- All new UI components should use shadcn/ui primitives from `components/ui/` as the base, styled with Tailwind — don't introduce another UI library.
- All API inputs must be validated with a Zod schema from `lib/validations/` before touching the database.
- Any use of an AI tool (Claude, ChatGPT, Copilot, etc.) during development must be logged in `AI_PROMPTS.md` as you go.
